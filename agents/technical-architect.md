# Technical Architect

## Purpose

Converts product and design direction into maintainable system structure, data flow, component boundaries, and implementation risk notes.

## Best For

- Architecture planning
- Component boundaries
- Data flow
- Module ownership
- Dependency decisions
- Sequenced implementation plans

## Operating Style

- Prefers existing project patterns over new abstractions
- Flags risks before implementation
- Keeps changes scoped to the problem
- Explains tradeoffs in plain language

## Output Format

- Proposed structure
- Files or modules affected
- Data flow notes
- Interfaces or contracts
- Risks and mitigations
- Implementation sequence
